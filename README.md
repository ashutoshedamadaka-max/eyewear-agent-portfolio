# Eyewear Recommendation Agent — "Specs"

A full-stack conversational AI agent that recommends eyewear from a 100-product Lenskart catalog. Backend on Vercel (Python/FastAPI), frontend on Lovable (React/TS), evals via Python harness with 10 quality metrics.

> **Built as a PM portfolio piece** demonstrating: agent design, RAG patterns, prompt engineering, voice/personality design, eval methodology, and full-stack deployment.

**🔗 Live demo:** [your-app.lovable.app] *(deploy via DEPLOYMENT_GUIDE.md)*
**🔗 Backend:** [your-backend.vercel.app/api/health]

| Endpoint | Method | Purpose |
|---|---|---|
| `/api/health` | GET | Sanity check — returns catalog size and key status |
| `/api/chat` | POST | Main agent endpoint — `{message, history}` → `{reply, recommended_ids, intent, path}` |
| `/api/catalog` | GET | Returns full product catalog |
| `/api/eval-results` | GET | Returns the latest eval_results.json |

Built-in protections:
- **Rate limit:** 30 requests per IP per hour (in-memory)
- **CORS:** open during development, lock to your Lovable domain in production
- **Error handling:** graceful degradation, never returns OpenAI errors directly

## Agent Architecture

```
                User message
                     │
                     ▼
            ┌────────────────┐
            │  Classifier    │  LLM #1: new_search? followup? smalltalk?
            └───────┬────────┘
                    │
       ┌────────────┼────────────┐
       ▼            ▼            ▼
   followup    new_search    smalltalk
       │            │            │
       │     ┌──────▼──────┐    │
       │     │  Intent     │    │  LLM #2: extract structured needs
       │     │  Extractor  │    │
       │     └──────┬──────┘    │
       │            │            │
       │       ready=true?      │
       │       ┌────┼────┐       │
       │       ▼         ▼       │
       │   recommend  clarify    │
       │       │         │       │
       ▼       ▼         ▼       ▼
   FOLLOWUP RECOMMEND  CLARIFY  CLARIFY
   (LLM #4)  (filter +  (LLM #3)
              LLM #3)
```

### Three conversation paths

In v2 every user message went through "extract intent → recommend or clarify." That broke when users asked follow-ups. The classifier-first architecture distinguishes:

- **new_search** — fresh shopping intent (extracts needs, then clarifies or recommends)
- **followup** — questions about products already on screen ("does it come in tortoise?")
- **smalltalk** — greetings, thanks, off-topic chitchat

This is the kind of conversational UX detail that separates a demo from a real product.

## The Voice Spec

Personality is a product decision. Specs's voice is codified in a `VOICE_SPEC` constant referenced by all system prompts:

> *"Quirky expert vibe. Knows frames the way a sommelier knows wine, but doesn't show off. Light dry humor when natural. Has opinions and shares them. Conversational, not corporate. Use contractions. Strictly avoid: 'I'd be happy to help', 'Great question', 'Based on your needs', emojis."*

A deterministic banned-phrase check in the eval harness fails any reply that drifts back into corporate-AI speak.

## Eval Methodology — 10 metrics across 13 test cases

### Rule-based (deterministic)
1. **Catalog Adherence** — every product_id must exist in the real catalog
2. **Budget Adherence** — recommendations within stated budget
3. **Category Adherence** — Eyeglasses request → Eyeglasses recs only
4. **Brand Adherence** — when brand named, only that brand
5. **Banned-Phrase Check** — deterministic personality drift detector
6. **Clarification Behavior** — agent asks when info missing

### LLM-as-judge (semantic)
7. **Use-Case Fit** — does the recommendation actually fit (1–5)
8. **Conversation Quality** — warmth, conciseness, helpfulness (1–5)
9. **Personality Consistency** — does this sound like Specs? (1–5)
10. **Follow-up Handling** — does the agent stay in context? (1–5)

## Failure modes anticipated and addressed

| Failure | Mitigation |
|---|---|
| Hallucinating products | Catalog grounding + ID-based eval check |
| Recommending over budget | Pre-filter step + budget eval |
| Wrong category | Pre-filter + category adherence eval |
| Endless clarifying questions | "Ready to recommend" needs only 2 fields |
| Bad JSON from extraction | OpenAI JSON mode + try/except fallback |
| Empty filter result (impossible budget) | Honest "couldn't find" reply, no hallucination |
| Personality drift | Centralized VOICE_SPEC + banned-phrase check + LLM judge |
| Context loss after recommendation | Classifier routes follow-ups to dedicated path |
| **API key exposure** | Server-side env var, never sent to browser |
| **Cost runaway** | Rate limit at 30 req/IP/hour + Vercel + OpenAI spend caps |

## Portfolio talking points

When you write the case study:

1. **Full-stack architecture decision.** Could've kept everything browser-side, but server-side API key handling is more secure and cost-controlled. Mentioned in resume as "designed and deployed Python/FastAPI backend on Vercel."

2. **Personality as a product decision.** Most AI demos sound the same. Specs has a written voice spec, banned phrases, and a deterministic check that catches drift.

3. **Classifier-first conversation routing.** Distinguishes new searches from follow-ups from smalltalk. This is why "does it come in tortoise?" works.

4. **Two new evals for two new failure modes.** Personality consistency and follow-up handling were added because v2 surfaced these gaps. New problems drive new evals.

5. **Pre-filter before LLM.** With 100 products, rule-based filtering is faster, cheaper, easier to debug than asking an LLM to filter. Senior PM thinking.

6. **Anti-hallucination by design.** Eval verifies catalog adherence automatically because the agent cites product_ids.

7. **Cost-conscious.** Rate limited, spend-capped, and gpt-4o-mini at ~₹3-7 per eval run.
